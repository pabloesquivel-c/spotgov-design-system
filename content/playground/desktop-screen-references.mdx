These screens test how the existing SpotGov system composes at full desktop
scale. The references contribute hierarchy, density, and rhythm. SpotGov keeps
its own sidebar, semantic tokens, procurement language, and accessible
interaction patterns.

Each example has a logical 1440 x 900 canvas. **Fit** keeps the complete
reference visible inside this article. **100%** preserves readable product
dimensions and lets the frame scroll inside its own presenter. Product screens
must reflow rather than copy this scaling behavior.

## Construction rules

The three examples share a 32px main-canvas inset and a 24px grid rhythm. They
reuse the existing `AppSidebar`, AlignUI type utilities, Remix Line icons, and
semantic SpotGov state tokens.

No primitive, token, global style, dependency, or production data contract was
added for these examples. The flat table and pipeline timeline are page
compositions, not new components in the product system.

## Bid operations dashboard

The dashboard is for prioritization, not generic metrics. Four focused widgets
cover urgency, bid readiness, reviews, and notes. The tall schedule region
keeps deadlines in the same scan path.

Try the review items, note checkboxes, schedule dates, and create action. The
empty schedule state is available on 25 July.

## Responsive inspection

The article presenter is tested at 1024, 1280, 1440, and 1728px. It must never
create page-level horizontal overflow. At narrower widths, Fit provides a
complete visual comparison and 100% keeps controls readable inside an internal
scroll region.

This presenter is not the responsive product contract. Production dashboards
reflow from three tracks to two, table toolbars wrap while table overflow stays
internal, calendar summaries move from four columns to two, and settings stack
when their rail and form no longer fit. The complete rules live in
`docs/screen-composition.md`.
